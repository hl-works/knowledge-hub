Agent Email
Boucles ouvertes

L'objet utile n'est pas le mail, c'est la boucle ouverte : un truc qui attend une action et qui n'est pas fait. Le mail est la matière première. Le produit central = un tableau de bord des boucles ouvertes + un digest quotidien qui en est l'instantané. Pattern de référence : l'agent factures fournisseurs — agent serveur autonome, pas une app qu'on ouvre à la main.

01 — Principe directeur

Trois règles dures, non négociables

Zéro dossier créé dans les boîtes Le rangement se fait dans des paniers du dashboard, pas en 100 sous-dossiers façon Fyxer.
Confiance binaire Rien de destructif ou de définitif (suppression dure, blocage) sans validation humaine. Mieux vaut afficher un item déjà réglé que d'en cacher un vrai.
Lecture seule d'abord, écriture réversible ensuite, destructif jamais en auto Même logique que l'agent factures : l'humain valide l'action finale.

Une action de ta part est attendue.

En attente v2

Tu as répondu, tu attends l'autre — suivi des relances qui slippent. Une fois la fermeture de boucle fiable.

Pour info

Rien à faire, juste savoir.

Bruit

Marketing / newsletters / com. Replié par défaut.

Suspect

Spam / phishing à vérifier.

04 — Détection

Bruit gratuit, spam/phishing en couches

Bruit — sans LLM, 0 token

Headers List-Unsubscribe, Precedence: bulk, signatures ESP (Mailchimp, Brevo, Sendinblue…). Attrape l'essentiel avant l'étage LLM et ne pollue pas le board.

Spam / phishing — en couches

  1. Auth (déterministe, gratuit, le meilleur signal) — lecture de Authentication-Results : SPF / DKIM / DMARC. Un mail qui se prétend « banque / fournisseur » et rate DMARC = phishing fort. C'est ce qu'Ionos ne te montre pas.
  2. Heuristiques — nom affiché ≠ domaine réel, domaines sosies (cobra.fr vs cobra-fr.com, homoglyphes), Reply-To divergent, texte de lien ≠ cible, langage « urgent, changez l'IBAN ».
  3. LLM en dernier juge, zone grise seulement — cher et faillible (injection possible via le contenu du mail), donc jamais sur tout.

Limite assumée : Ionos est le MTA de réception ; on ne touche la boîte qu'en IMAP, après livraison. On construit un deuxième avis qui re-juge et déplace, pas un portail qui bloque à l'entrée. Un vrai filtrage amont = re-router le MX → autre projet, hors périmètre.

05 — Gouvernance des écritures

Confiance × réversibilité

ActionRéversible ?Déclenchement
Classer en panier (base interne)ouiauto
Quarantaine (dossier « Suspect »)oui (récupérable)auto sur haute confiance
Sender blocké → Corbeilleouiauto (une fois blocké)
Supprimer + blocker un expéditeurajout blocklistmanuel, 1 tap dans le digest
Suppression dure (expunge)nonmanuel uniquement
Envoi de mailnonjamais auto, brouillon seulement

Le « blocage » = une blocklist maintenue par l'agent (adresses + domaines), pas un blocage Ionos (inexistant en IMAP). Centralisée sur les 8 boîtes, réversible, indépendante du provider. Décision manuelle une fois par expéditeur, puis auto. Spam ≠ phishing dans l'auto : le spam se quarantine agressivement (au pire une newsletter perdue) ; le phishing se quarantine + te notifie — jamais en silence, il peut usurper un contact réel. La blocklist est visible et éditable dans le dashboard.

06 — Sorties

Digest, dashboard, brouillons

  • Digest quotidien Telegram (matin) : nouvelles boucles + boucles qui vieillissent (« pas traité depuis X jours ») + suspects à valider avec boutons inline supprimer + blocker.
  • Dashboard web sous hl-consulting.tech/mail, protégé par mot de passe (StatiCrypt, comme la carto HVS) — pas public, il expose tes boucles ouvertes. Mobile-first, stack légère, cohérent avec le knowledge-hub.
  • Brouillons de réponse générés à la demande sur « À répondre », déposés dans les Drafts (Graph draft / IMAP APPEND). Jamais d'envoi automatique.

07 — Plan de dev

10 étapes, chacune avec un critère de succès exécutable

Étape 0Socle

Repo /opt/agents/email, chargeur de config (creds hors repo, chmod 600), schéma SQLite, logger à compteurs (lus / nouveaux / dédupliqués / classés / rejetés). Interface normalisée MailSource : list_since(), get_message(), move(), append_draft().

✅ Critère — selftest liste les 5 derniers mails d'UNE boîte Ionos, les écrit en base, 0 écriture dans la boîte.

Étape 1Ingestion multi-boîtes, lecture seule (6 boîtes)

Adaptateur IMAP (5 Ionos) + réutilisation du Graph cobra. Fetch incrémental (UIDVALIDITY/UIDNEXT), normalisation, dédup par fil (Message-ID / References / In-Reply-To).

✅ Critère — un run traite les 6 boîtes sans doublon, compteurs en log, 0 écriture.

Étape 2Pré-filtre déterministe (bruit + suspect), gratuit

Règles headers → panier Bruit. Checks Authentication-Results + heuristiques → score Suspect. Aucun LLM.

✅ Critère — sur hautefidelite-magazine.com, l'essentiel du bruit classé sans LLM, les DMARC-fail remontent en Suspect.

Étape 3Étage LLM : extraction de boucles + résumé

Sur le mail humain restant : classer le panier, extraire la boucle (action + owner moi/autre), résumer. Haiku pour le triage de masse, Sonnet/Opus pour la synthèse du digest.

✅ Critère — le « méga résumé » quotidien de cobra.fr seul est lisible et juste sur un échantillon relu à la main.

Étape 4Digest Telegram

Bot, digest matinal poussé : nouvelles boucles + vieillissantes + suspects.

✅ Critère — digest reçu chaque matin, contenu = état réel de la base.

Étape 5Dashboard web (hl-consulting.tech/mail)

Sous-chemin + auth, servi depuis le serveur. Mobile-first, board trié par âge (« À répondre » ouvert, « Bruit » replié), cases « fait », blocklist éditable.

✅ Critère — tu ouvres la page (après login) depuis ton tel, tu vois les boucles par ancienneté, tu coches « fait ».

Étape 6Actions d'écriture réversibles (quarantaine)

Déplacement auto vers « Suspect » (haute confiance) ; sender blocké → Corbeille. Rien de dur.

✅ Critère — un sender blocké voit ses mails déplacés au run suivant, récupérables.

Étape 7Actions validées + brouillons

Boutons inline supprimer + blocker dans le digest → maj blocklist. Brouillons de réponse à la demande sur « À répondre ».

✅ Critère — 1 tap ajoute à la blocklist et corbeille le fil ; un brouillon apparaît dans Drafts, jamais envoyé.

Étape 8Activation Gmail + adaptateur Hotmail

Gmail = activation config sur l'adaptateur IMAP générique (host + app password), pas de nouveau code. Hotmail = seul vrai dev restant (Graph perso / IMAP-OAuth2).

✅ Critère — les 8 boîtes tournent dans le même run ; activer Gmail n'a demandé que de la config.

Étape 9Autonomie (cron) + garde-fous

Timer systemd (comme factures), kill switch, alerte si un adaptateur tombe.

✅ Critère — tourne seul chaque matin, tu ne l'ouvres que pour valider.

08 — Stack & build

Ce sur quoi ça tourne

Python · imapclient/imaplib (IMAP) + SDK Graph existant · SQLite (état, comme le processed-tracker factures) · python-telegram-bot · API Claude (Haiku triage / Sonnet-Opus synthèse & brouillons) · dashboard Flask/FastAPI minimal + HTML statique, zéro framework lourd · systemd timer pour l'autonomie.

À brancher au build (jamais dans le repo)

  • App password Gmail (2FA)
  • OAuth Hotmail
  • Token bot Telegram
  • Domaines / hosts IMAP exacts de hl-media et hl-holding
